WildCart — Project Documentation

Full-stack Laravel e-commerce application  |  Laravel 11 · Blade · Bootstrap 5 · AdminLTE 3

1. Project Overview

WildCart is a fully functional e-commerce web application built with the Laravel PHP framework. It has two distinct areas:

  • Customer-facing shop — browsing products, cart, checkout (guests & logged-in users), order history, reviews, contact form.
  • Admin panel — managing products, categories, orders, users, contact messages, and viewing activity logs.

The frontend uses Bootstrap 5 for the shop and AdminLTE 3 (Bootstrap 4) for the admin. All dynamic interactions (cart updates, live search, status changes) use AJAX via jQuery — no full-page reloads.

Why user_id is nullable on orders

The site allows guest checkout — a user can place an order without an account. In that case user_id is NULL and a guest_email is stored instead. This was added via a dedicated migration after the initial table creation:

// migration: make_order_user_nullable_add_guest_email
$table->foreignId('user_id')->nullable()->change();
$table->string('guest_email')->nullable()->after('user_id');

The role column on users

Rather than a separate roles table, users have a simple role string column (value: "user" or "admin"). The User model exposes a helper method:

public function isAdmin(): bool
{
    return $this->role === 'admin';
}

Accessor — average_rating on Product

The Product model has a computed attribute that calculates the average review rating on the fly, without storing it in the database:

public function getAverageRatingAttribute()
{
    return $this->reviews()->avg('rating') ?? 0;
}

In Blade templates you can then access it as $product->average_rating — Laravel automatically maps getXxxAttribute() methods to property access.

Route Model Binding

Laravel automatically looks up a model from the database based on the URL parameter. For example:

Route::get('/shop/{product:slug}', [ProductController::class, 'show']);

The :slug part tells Laravel to find the Product where slug = [url segment] instead of the default id. If nothing is found, Laravel returns a 404 automatically — no manual findOrFail needed.

Route::resource

Instead of registering 7 separate routes for CRUD, Route::resource('products', ProductController::class) automatically creates: index, create, store, show, edit, update, destroy — following REST conventions.

4. Middleware & Access Control

Middleware is code that runs before a request reaches a controller. The project uses two middleware stacked together for admin routes:

  • auth — built-in Laravel middleware, redirects unauthenticated users to the login page.
  • admin — custom middleware in app/Http/Middleware/AdminMiddleware.php.
public function handle(Request $request, Closure $next): Response
{
    $user = Auth::user();
    if (!Auth::check() || !$user->isAdmin()) {
        abort(403, 'Access denied.');
    }
    return $next($request);
}

If the user is not logged in or is not an admin, it immediately returns a 403 Forbidden response. Otherwise it calls $next($request) to pass the request further down to the controller.

5. AppServiceProvider — Bootstrap & Shared Data

app/Providers/AppServiceProvider.php is a special file that runs once on every request during Laravel's boot phase. We use it for three things:

Global pagination style

Paginator::useBootstrapFive();

This makes all ->paginate() calls render Bootstrap 5 pagination links by default, site-wide.

Sharing nav links with every view

View::share('mainNavLinks', [
    ['label' => 'Home',    'route' => 'home',       'icon' => 'fa-home',     'active' => ['home']],
    ['label' => 'Shop',    'route' => 'shop.index',  'icon' => 'fa-store',    'active' => ['shop.index', 'shop.show']],
    ['label' => 'Contact', 'route' => 'contact',     'icon' => 'fa-envelope', 'active' => ['contact']],
]);

View::share() injects a variable into every Blade template automatically. The navbar and footer both loop over $mainNavLinks — so adding or removing a nav item only requires editing this one place.

Listening to auth events

Laravel fires built-in events when users register, log in, and log out. We hook into those to automatically create activity log entries:

Event::listen(Login::class, function (Login $event) {
    ActivityLog::log('user_login', "User '{$event->user->name}' logged in", $event->user->id);
});

This means we never have to manually log these events inside controllers — it happens automatically.

6. Controllers

HomeController

Loads the home page with three pieces of data:

  • Featured products — the 8 most recently added active products.
  • Categories — only categories that have at least one active product (uses withCount + having).
  • Total product count — displayed in the hero section.
$categories = Category::withCount(['products' => fn ($q) => $q->where('active', true)])
    ->having('products_count', '>', 0)
    ->get();

withCount adds a virtual products_count column to each category result. having filters by that count — this is SQL's HAVING clause, used because you can't use WHERE on aggregate columns.

ProductController

index() — Shop listing with filters

Builds a query dynamically based on what filters are present in the URL:

$query = Product::with('category')->where('active', true);

if ($search = $request->get('search'))      { /* name/description LIKE search */ }
if ($categories = $request->get('categories')) { $query->whereIn('category_id', ...); }
if ($request->filled('price_min'))          { $query->where('price', '>=', ...); }
if ($sort = $request->get('sort'))          { /* match expression for ordering */ }

$products = $query->paginate(12)->withQueryString();

withQueryString() preserves all current URL parameters (search, filters, sort) in the pagination links — so clicking "next page" doesn't lose your filters.

When the request is AJAX (sent by JavaScript), it returns JSON with rendered HTML partials instead of a full page:

if ($request->ajax()) {
    return response()->json([
        'html'       => view('partials.product-grid', compact('products'))->render(),
        'pagination' => view('partials.pagination', compact('products'))->render(),
    ]);
}

search() — Live navbar search

Returns a JSON array of up to 6 matching products. Called by JavaScript as the user types (with debouncing to avoid too many requests). Requires at least 2 characters before querying.

show() — Product detail page

Uses eager loading to fetch related data in as few queries as possible:

$product->load(['category', 'reviews' => fn($q) => $q->with('user')->latest()]);

load() is eager loading after the model is already fetched. The callback fn($q) => $q->with('user')->latest() constrains the reviews relationship — it loads reviews with their authors, sorted newest-first.

CartController

The cart is entirely session-based — no database table is involved. The cart is stored as an associative array in the PHP session:

// Structure: ['product_id' => quantity, ...]
// e.g.: [3 => 2, 7 => 1]  means: product #3 qty 2, product #7 qty 1
session()->get('cart', []);
session()->put('cart', $cart);

This means the cart works for guests too (no login needed), and persists across pages as long as the session is alive.

The add(), update(), and remove() methods all return JSON responses — they are called via AJAX. After each change, the response includes the new cart count, subtotals, and totals so the page can update without a reload.

Shipping is calculated inline everywhere the cart is read:

$shipping = $subtotal >= 50 ? 0 : 5.99; // free shipping over $50

OrderController (customer-side)

create() — Checkout page

Reads the cart from session and passes the items + totals to the checkout view. If the cart is empty, redirects back to the cart page.

store() — Placing an order

This is the most important method for the order flow. Here's what happens step by step:

  1. Validates shipping fields. If the user is a guest, also validates guest_email.
  2. Fetches product models from the database to get current prices.
  3. Calculates subtotal, shipping, and total.
  4. Creates the Order record (with user_id = null for guests).
  5. Loops through the cart and creates one OrderItem record per product.
  6. Decrements stock for each purchased product: $products[$productId]->decrement('stock', $qty)
  7. Clears the cart from session: session()->forget('cart')
  8. Redirects logged-in users to their order page; guests back to home with a flash message.

Prices are snapshotted in OrderItem.price at the time of purchase. This means if the product price changes later, old orders still show the correct historical price.

Admin\OrderController — Order Status State Machine

Orders cannot jump to any status arbitrarily. The allowed transitions are enforced in code:

public static function allowedTransitions(): array
{
    return [
        'pending'    => ['processing', 'cancelled'],
        'processing' => ['shipped',    'cancelled'],
        'shipped'    => ['delivered',  'cancelled'],
        'delivered'  => [],   // locked — no further changes
        'cancelled'  => [],   // locked — no further changes
    ];
}

When an admin submits a status change via AJAX, updateStatus() checks if the new status is in the allowed list for the current status. If not, it returns a 422 JSON error.

Two important side effects happen on status change:

  • Stock restore on cancel — if the new status is cancelled, the quantity of every order item is added back to the product's stock.
  • Email notification — an email is sent to the customer (or guest email). This is wrapped in a try-catch so a mail failure doesn't break the response.

ReviewController

Reviews are submitted via AJAX (POST to /shop/{product}/reviews). The controller:

  1. Validates: rating 1–5 (required), body 10–1000 chars (required).
  2. Checks the database for an existing review by the same user for the same product — one review per user per product is enforced here.
  3. If no duplicate, creates the review and returns JSON with the new review data and the updated average rating.

The page doesn't reload — JavaScript receives the JSON and prepends the new review to the list and updates the star display.

ProfileController — Account Deletion

Before deleting an account, the controller checks for active orders:

$activeOrders = Order::where('user_id', $user->id)
    ->whereIn('status', ['pending', 'processing', 'shipped'])
    ->count();

if ($activeOrders > 0) {
    return Redirect::route('profile.edit')->withErrors(
        ['active_orders' => "You have {$activeOrders} active order(s)..."],
        'userDeletion'
    );
}

The error is sent in the userDeletion named error bag — this allows the profile page to re-open the delete modal automatically and show the warning without affecting any other form on the page.

ContactController

Validates and saves the contact message to the database first, then tries to send a notification email. The email is wrapped in try-catch — even if the mail service is down, the message is already safely stored in the DB.

7. Authentication — Laravel Breeze

Authentication (register, login, logout, password reset, email verification) is provided by Laravel Breeze — a minimal auth scaffolding package. It generates all the necessary controllers, views, and routes automatically. The generated routes live in routes/auth.php which is included at the bottom of web.php.

Breeze uses session-based authentication — when a user logs in, their user ID is stored in the session and Laravel rehydrates the Auth::user() on every subsequent request.

Key things Breeze provides:

  • Secure password hashing (bcrypt, automatic via the 'password' => 'hashed' cast in the User model)
  • CSRF protection on all forms (the @csrf directive in Blade)
  • Remember me token support
  • Email verification flow

8. Blade Templating

Blade is Laravel's templating engine. All view files live in resources/views/ and use the .blade.php extension.

Layout inheritance

The shop and admin have separate master layouts. All shop pages extend a shared layout:

{{-- In any shop page --}}
@extends('layouts.app')

@section('title', 'Shop')

@section('content')
  <!-- page content goes here -->
@endsection

The layout file (layouts/app.blade.php) has @yield('content') and @yield('title') placeholders which get filled in by each page. This avoids repeating the navbar/footer on every file.

Partials and includes

Reusable pieces of HTML are extracted into partials and included with @include:

@include('partials.navbar')
@include('partials.footer')

9. AJAX Interactions

Several user-facing actions are handled via AJAX so the page doesn't reload. All AJAX calls use jQuery's $.ajax() and send the Laravel CSRF token in the header:

Debouncing (live search)

When the user types in the search box, we don't want to fire a request on every single keystroke. Debouncing delays the request until the user has stopped typing for a set time (300ms):

let searchTimeout;
$('#navSearchInput').on('input', function () {
    clearTimeout(searchTimeout);
    searchTimeout = setTimeout(() => {
        // fire AJAX search
    }, 300);
});

10. Activity Logging

The ActivityLog model has a static log() helper method that makes logging a one-liner anywhere in the codebase:

ActivityLog::log('cart_add', "Added '{$product->name}' (qty: {$quantity}) to cart");
ActivityLog::log('order_created', "Order #{$order->id} created, total: {$total}");
ActivityLog::log('review_posted', "Posted review for product '{$product->name}'");

The method automatically reads the current logged-in user and the client's IP address:

public static function log(string $action, string $description, ?int $userId = null): self
{
    return self::create([
        'user_id'     => $userId ?? Auth::id(),
        'action'      => $action,
        'description' => $description,
        'ip_address'  => request()->ip(),
    ]);
}

Three events (register, login, logout) are logged automatically via Laravel event listeners in AppServiceProvider, so no controller code is needed for those.

11. Email Notifications

Emails are sent using Laravel's Mailable class system. The project has two mailables:

  • App\Mail\OrderStatusMail — sent on every order status change
  • App\Mail\ContactMail — sent to the admin when someone submits the contact form

How a Mailable works

class OrderStatusMail extends Mailable
{
    public function __construct(public Order $order, public ?string $cancelReason = null) {}

    public function envelope(): Envelope
    {
        return new Envelope(subject: 'Your WildCart Order #' . $this->order->id . ' — Status Update');
    }

    public function content(): Content
    {
        return new Content(view: 'emails.order-status');
    }
}

The constructor uses PHP 8 constructor property promotion (public Order $order) — this declares the property and assigns it in one line. The $order and $cancelReason are automatically available inside the emails/order-status.blade.php view.

Sending the email:

$email = $order->user?->email ?? $order->guest_email; // null-safe operator
Mail::to($email)->send(new OrderStatusMail($order, $cancelReason));

$order->user?->email uses the null-safe operator (?->) — if $order->user is null (guest order), it returns null instead of crashing. Then ?? falls back to guest_email.

The entire send is wrapped in try-catch so a mail service failure logs an error but doesn't break the order status update response.

During development, mail is tested with Mailtrap — a fake SMTP inbox that captures emails without actually sending them.

12. Admin Panel

The admin panel lives under /admin and is built with AdminLTE 3 (a Bootstrap 4 admin template). Its controllers are in a separate namespace: App\Http\Controllers\Admin\.

Dashboard

Displays four summary stats, recent orders, and recent activity:

$stats = [
    'users'    => User::count(),
    'products' => Product::count(),
    'orders'   => Order::count(),
    'revenue'  => Order::where('status', '!=', 'cancelled')->sum('total'),
];

Revenue excludes cancelled orders using a simple where filter before sum().

Product management

Uses Route::resource which maps to a standard set of controller methods: index, create, store, edit, update, destroy. Product images are uploaded to storage/app/public/ and served via a symlink that php artisan storage:link creates.

User management

Admins can toggle another user's role between user and admin via AJAX. A safeguard prevents an admin from deleting or demoting their own account.

Order management filters

The admin orders list supports filtering by status and searching by customer name/email. The search uses a whereHas query — it queries through the relationship:

$query->whereHas('user', fn($q) =>
    $q->where('name', 'like', "%{$search}%")
      ->orWhere('email', 'like', "%{$search}%")
);

whereHas translates to a SQL EXISTS subquery — it only returns orders where the related user matches the search term.

13. Guest Checkout Flow

One of the more interesting design decisions is that checkout does not require an account. Here is how this is handled:

Guest visits site → adds products to cart (session storage, works without login) → goes to /checkout → fills in shipping info + provides email address → POST /orders → user_id = NULL → guest_email = submitted email → order created, stock decremented → redirected to home with flash message → receives email notification on status changes

When the admin views a guest order, the code safely handles the missing user with the null-safe operator:

$email = $order->user?->email ?? $order->guest_email;

In Blade, null-safe access is done with optional() or direct null checks before accessing user properties.

14. Eager Loading — Preventing N+1 Queries

This is an important performance concept in Laravel. Consider loading 10 orders and then showing each order's user name. Without eager loading:

// BAD — N+1 problem: 1 query for orders + 1 query per order for its user = 11 queries
$orders = Order::all();
foreach ($orders as $order) {
    echo $order->user->name; // each access fires a new SQL query
}
// GOOD — eager loading: 2 queries total (one for orders, one for all users)
$orders = Order::with('user')->get();

Throughout the project, ->with() is used whenever related data is needed in a view. For example:

Order::with('user')->orderByDesc('created_at')->paginate(15);
$order->load('user', 'items.product'); // dot notation loads nested: items AND each item's product

15. Form Validation

Laravel validates request data with $request->validate([]). If validation fails, Laravel automatically redirects back with the errors in the session — no manual error handling needed.

$request->validate([
    'shipping_name'    => 'required|string|max:255',
    'shipping_address' => 'required|string|max:255',
    'guest_email'      => 'required|email|max:100', // only added for guests
]);

In Blade, errors are displayed using $errors:

@error('shipping_name')
    <div class="text-danger">{{ $message }}</div>
@enderror

For AJAX endpoints (like reviews and cart), the controller returns a JSON response with status 422, and JavaScript handles displaying the error message.
